The data hierarchy

Sets, projects, banks, patterns, parts. The Octatrack's data structure is the single most important thing to understand before you touch a sequencer page — and the source of most newcomer confusion.

The Octatrack does not have a "song" or a "preset". It has a layered structure where data lives at five different scopes: the card, the set, the project, the bank, and the pattern/part. Knowing which scope a piece of data belongs to is how you avoid the two most common Octatrack disasters: losing work, and accidentally overwriting something elsewhere on the machine.

Card

The Compact Flash card is the root of everything. Format it FAT32 and the Octatrack creates a top-level folder structure on first boot. A single card can hold dozens of sets, each one a self-contained workspace. Cards are interchangeable — taking one out and inserting another is how you switch between, say, a live-set CF and a studio-experiments CF without having to manage projects within a single card.

Set

A set is a folder on the card. It contains an Audio Pool (the shared sample library, see chapter 7) and one or more projects. The set is the boundary between which samples a project can see — projects within the same set share the Audio Pool; projects in different sets do not.

Practical use of sets: one set per album, one per gig, one per studio session. The Audio Pool grows over time within a set; starting a fresh set keeps things tidy.

Project

The project is the "song" container, though the metaphor is loose. It holds:

  • 16 banks (A–P), each containing 16 patterns and 4 parts
  • Project-wide settings — sync source, MIDI config, audio routing, memory configuration
  • The Arranger — up to 8 arrangements per project, chaining patterns into long-form structures
  • References to samples in the Audio Pool, plus per-project sample slots (Static + Flex slot lists)

Loading a project loads all of the above into RAM. Saving a project writes the current RAM state back to the card. Unsaved changes are lost on power-off — the OT does not auto-save.

Bank

A bank is 16 patterns plus 4 parts, identified by a letter A–P. Banks are independent: switching from bank A to bank B replaces the patterns and parts in active memory. Banks A–P are accessed by holding BANK and pressing the corresponding trig key.

The 4 parts within a bank are shared across that bank's 16 patterns. Any pattern can be assigned to any of the 4 parts — a pattern's "instrument set" comes from its assigned part.

Pattern

A pattern is the sequence: the trigs, parameter locks, microtiming offsets, conditional trigs, and per-track pattern length. Patterns are 1 to 64 steps, with per-track lengths possible for polyrhythmic sequences.

A pattern references a part — it does not contain the instrument data itself. Two patterns assigned to the same part will play through the same set of machines and samples; changing the part affects both.

Part

The part is the instrument container. It holds:

  • Per-track machine assignments (Static, Flex, Thru, Neighbor, Pickup, Master)
  • Per-track sample slot assignments (which sample each Flex/Static track plays)
  • Per-track parameter base values for all five pages (PLAYBACK, AMP, LFO, FX1, FX2)
  • Per-track scene assignments for both ends of the crossfader
  • Per-track MIDI track data (channel, port, base note, etc.)

A pattern + its part = the complete musical statement at that moment. Saving a part is separate from saving a pattern, and worth doing whenever you've made changes you want to keep.

Warning · the unsaved-part problem

The most common Octatrack frustration: making careful per-track adjustments, switching to a different pattern, returning to the original — and finding all the adjustments wiped. The cause is almost always an unsaved part. Patterns hold their changes in RAM, but switching parts reloads the part from disk if the in-memory copy hasn't been saved. FUNCTION+YES on the part page commits the part to disk; do this often.
